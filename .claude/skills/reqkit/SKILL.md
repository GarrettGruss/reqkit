# reqkit

`reqkit` (implemented in `src/reqkit/`, installed as the `reqkit` CLI via `uv run reqkit ...`)
mints, parses, and traces a pseudo-XML tag scheme described in `docs/spec.md`. Tags are not a
full XML document — they're small, self-contained XML elements (`<rq-req ...>`, `<rq-trace ...>`)
that appear inline inside otherwise-arbitrary text: markdown, source code comments, anywhere.

## Tag shape

```
<rq-req id="<hex>" subtype="<type>" category="<cat>" parent_id="<id>" parent_rel="<rel>">body</rq-req>
<rq-trace parent_id="<id>" parent_rel="<rel>">body</rq-trace>
```

- All attributes are optional except `id` on `rq-req` (auto-generated by `mint` if omitted).
- Self-closing form (`<rq-req ... />`) is valid wherever there's no body.
- `subtype` (on `rq-req`) must be one of: `req`, `fr`, `nfr`, `br`, `ur`, `tr`, `sr`.
- `parent_rel` (on either tag) must be one of: `satisfy`, `derive`, `trace`, `verify`, `refine`
  (see `docs/spec.md` for the full semantics of each relation).
- `rq-trace` has no `id` — it exists only to link back to a `parent_id`, e.g. a requirement
  referencing the code or test that implements it.
- Both validators live on the pydantic models in `src/reqkit/types.py` and raise on
  construction, so an invalid `subtype`/`parent_rel` anywhere (mint, parse) is caught immediately.

## Commands

### `mint` — generate new requirement tag(s)

```bash
uv run reqkit mint [--subtype <type>] [--category <cat>] [--parent_id <id>] [--parent_rel <rel>] [--qty <n>]
```

Prints each newly minted `<rq-req>` tag to stdout (id auto-generated via the `RQ_ID_STRATEGY` env
var, hex by default).

### `parse` — extract tags from a single file

```bash
uv run reqkit parse <file>
```

Runs `parse_str` (in `src/reqkit/parser.py`) against the file contents and prints each
successfully-parsed `<rq-req>`/`<rq-trace>` tag. Tags are extracted individually with a regex (the
file as a whole is not valid XML); only the opening tag's attributes are parsed as XML — the body
is kept as raw text so it can contain unescaped `&`/`<` (e.g. real source code). Tags that fail XML
parsing or fail pydantic validation are skipped, each with a `Skipping tag that failed ...`
message printed to console — the file does not need to be all-or-nothing.

### `trace` — recursive compliance report

```bash
uv run reqkit trace --source <file-or-dir> [--target <file-or-dir>] [--full | --partial | --minimal]
```

- `--source` is required; `--target` is optional. Both accept a single file or a directory, which
  is walked recursively, parsing every readable text file (binary files are skipped, not errored).
- `--source`/`--target` are purely semantic labels (typically: source = requirements doc(s),
  target = code to trace against) — under the hood they're just two parse passes whose results
  get merged into one pool of objects before tracing. If `--target` is omitted, only `--source` is
  walked.
- Output verbosity (default `--partial`):
  - `--full` — requirement body + trace body
  - `--partial` — requirement body only, no trace body
  - `--minimal` — no body text at all (attributes only)
- Builds a `{parent_id: [objects]}` map (`Tracer.build_map`) and renders a report
  (`Tracer.construct_report`) that, for every requirement, prints:
  - **non-compliant** — no traces point at it
  - **partially-compliant** — has traces, but none with `parent_rel` `satisfy`/`verify`
  - **compliant** — has at least one `satisfy`/`verify` trace
  - nested `derive`/`refine` child requirements indented underneath their parent
  - a footer: total requirements, total traces, and orphaned traces (traces whose `parent_id`
    doesn't resolve to any known requirement — including a dangling/typo'd id).

## When to reach for this skill

- Creating a new requirement: use `mint` rather than hand-writing an id (ids should come from the
  configured generation strategy to stay collision-resistant).
- Checking whether a file's tags are well-formed before committing it: run `parse` and look for
  `Skipping tag that failed ...` lines.
- Assessing compliance/drift between a requirements doc and its implementation: run `trace` with
  `--source` pointed at the doc(s) and `--target` at the code, and read the non-compliant /
  partially-compliant / orphaned counts in the footer.
